# ADR-002: httpx AsyncClient with Per-Plugin Token Injection for Device REST Calls

**Status:** Accepted  
**Date:** 2026-06-17  
**Deciders:** Perceptua  

---

## Context

autonomon plugins drive nomon devices by calling the `nomothetic` REST API
(HTTPS on port 8443). Three aspects of this connection required explicit
decisions:

1. **TLS verification:** Device certs are self-signed (nomothetic ADR-001).
   Standard TLS verification rejects them. Options: bundle the device cert,
   use a system CA bundle, or skip verification.

2. **Authentication:** All device endpoints require a Bearer JWT token.
   The token (`NOMON_PLUGIN_TOKEN`) is provided to the plugin at startup via
   environment variable by the `AutonomyPluginManager` (nomothetic ADR-019).
   Options: pass the token to each layer impl, inject it at the HTTP client
   level, or use a shared singleton.

3. **HTTP client library:** `requests` (sync), `httpx` (async + sync), or
   `aiohttp` (async-only). The `asyncio`-based pipeline architecture requires
   non-blocking I/O in perception and action layers.

## Decisions

### D1 — Use `httpx.AsyncClient` with `verify=False`

All HTTP calls from autonomon plugins use `httpx.AsyncClient(verify=False)`.
TLS peer certificate verification is disabled for device connections.

### D2 — Inject the token via default headers at client construction

The Bearer token is injected once, at `AsyncClient` construction, via
`headers={"Authorization": f"Bearer {token}"}`. Individual layers and methods
do not handle token plumbing.

```python
client = httpx.AsyncClient(
    base_url=device_url,
    verify=False,
    headers={"Authorization": f"Bearer {token}"},
    timeout=5.0,
)
```

### D3 — One `AsyncClient` instance per plugin run, shared across layers

The client is created in the plugin's `cli.py` entry point (or a shared factory)
and passed to each layer implementation as a constructor argument. It is closed
in the plugin's `finally` block.

```python
async with httpx.AsyncClient(...) as client:
    pipeline = Pipeline(
        perception=UltrasonicPerception(client),
        world_model=ObstacleWorldModel(),
        planner=AvoidancePlanner(),
        action=VehicleAction(client),
    )
    await pipeline.run()
```

### D4 — `asyncio.wait_for` for per-request timeouts, not retry loops

Network calls inside perception and action layers use:

```python
resp = await asyncio.wait_for(
    client.get("/api/hat/ultrasonic"),
    timeout=1.0,
)
```

On `asyncio.TimeoutError` or `httpx.RequestError`, the layer logs the failure
and continues its poll loop. It does not retry automatically. Back-pressure
from the bounded queue naturally slows the pipeline when hardware is unavailable.

## Rationale

**Why `verify=False` instead of bundling the device cert?**  
Device certs are auto-generated by nomothetic on first run and stored in `.certs/`
(gitignored). There is no CA infrastructure to distribute certs to plugins.
Bundling requires copying the cert file to the plugin environment on every
device, coupling plugin deployment to the device cert lifecycle. `verify=False`
is acceptable because: (a) plugins always run on the same device as nomothetic
or on a trusted LAN segment, (b) the JWT token provides authentication
independently of TLS peer auth, and (c) the threat of MITM on a closed robot
LAN is low. This matches nomothetic ADR-001. See also: nomothetic sentinel
check AU6 — `verify=False` must never be used for external services.

**Why token injection at the client level?**  
Layer implementations (`PerceptionBase`, `ActionBase` etc.) must not know about
authentication. Injecting the token into the shared `AsyncClient` instance keeps
auth out of the layer contracts. If the token is replaced (future: token refresh),
only the client construction site changes.

**Why one client per plugin run, not per layer or per request?**  
`httpx.AsyncClient` manages a connection pool. Creating a new client per layer
wastes connections and inhibits TCP/HTTPS keep-alive reuse between perception
polling and action commanding. Creating a new client per request adds TLS
handshake overhead (~50–100 ms on a Pi Zero 2W) on every poll cycle.

**Why `httpx` over `requests` or `aiohttp`?**  
`requests` is synchronous. Calling it from an `async def` layer blocks the
asyncio event loop, stalling queue processing and causing back-pressure
throughout the pipeline. `asyncio.to_thread(requests.get, ...)` works but adds
a thread-pool entry per sensor poll — expensive at high poll rates.

`aiohttp` is async-only and would add a dependency with no advantage over
`httpx`. `httpx` supports both sync and async clients (useful for tests that
don't need a running event loop), integrates cleanly with `pytest-httpx` for
mocking, and is already used as an optional dependency by `nomothetic` itself
(`[central]` extra).

**Why `asyncio.wait_for` instead of httpx's built-in `timeout`?**  
httpx's `timeout` parameter applies per-phase (connect, read, write). Using
`asyncio.wait_for` with a wall-clock deadline is simpler to reason about in the
context of a poll loop: "if this sensor read takes longer than 1 s, skip it and
try again next cycle." Both approaches are correct; `asyncio.wait_for` is more
explicit in the layer contract.

## Testing

Layer implementations accept an injectable `httpx.AsyncClient` argument:

```python
class UltrasonicPerception(PerceptionBase):
    def __init__(self, client: httpx.AsyncClient) -> None:
        self._client = client
        self._stop = asyncio.Event()
```

Tests pass a `pytest-httpx` mock client or an `unittest.mock.AsyncMock`:

```python
async def test_emits_perception_event(mock_client):
    mock_client.get.return_value = Response(200, json={"distance_cm": 18.4})
    perception = UltrasonicPerception(mock_client)
    q: asyncio.Queue = asyncio.Queue()
    task = asyncio.create_task(perception.run(q))
    ev = await asyncio.wait_for(q.get(), timeout=1.0)
    assert ev["sensor_type"] == "ultrasonic"
    await perception.stop()
    await task
```

No real device or network required. No `verify=False` consequences in tests.

## Constraints

- `verify=False` is permitted **only for device-local connections** (host is
  `NOMON_DEVICE_URL`, always a Pi on the local LAN or loopback).
- If autonomon ever calls a non-device URL (e.g. a cloud LLM API), a separate
  `httpx.AsyncClient(verify=True)` must be used for that call.
- `NOMON_PLUGIN_TOKEN` must never appear in log output. Log the URL and status
  code, not the Authorization header.
